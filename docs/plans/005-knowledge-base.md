# Plan: Knowledge Base (Project Context Files)

**Status**: Draft
**Date**: 2026-04-10
**Author**: Jonas + Claude

## Overview

Add a knowledge base feature that lets users attach document files (.md, .txt, .json, .yml) to their chat. Files exist at two scopes: **project-level** (available across all conversations) and **conversation-level** (scoped to a single chat session). The system auto-generates YAML frontmatter (name, description, tags) for each uploaded file. A lightweight catalog of frontmatter metadata is injected into every LLM prompt, and the agent retrieves full file contents on demand via a `read_knowledge_file` tool. This follows the progressive disclosure pattern already used by the skills system.

### Phase 7: UI/UX specifications `Not Started`

This phase runs in parallel with Phases 5-6. All items below must be implemented before the feature ships.

#### Layout and responsiveness

- [ ] Desktop (>=1024px): collapsible sidebar (280px width) to the left of the chat column. Toggle button in the chat header. Chat column retains its current `max-width: 720px` centered layout when sidebar is closed; shifts right when sidebar is open.
- [ ] Tablet (768-1023px): sidebar as an overlay drawer, slides in from the left with a scrim backdrop (50% black). Chat area remains full width underneath.
- [ ] Mobile (\<768px): sidebar as a full-screen overlay. Toggle via header button. Close on file selection or explicit close button.
- [ ] Sidebar open/close: slide transition, 200ms ease-out. Overlay scrim fades 150ms.

#### Empty states

- [ ] Sidebar empty (project scope): illustration or icon + "No project files yet" + "Upload a document to give the assistant persistent context across conversations." + Upload CTA button.
- [ ] Sidebar empty (conversation scope): "No files attached to this conversation" + "Upload or drag a file here, or use @ in the chat input."
- [ ] Editor empty content: placeholder text appropriate to file type ("Start writing markdown..." / "Paste JSON here...").

#### Upload feedback flow

- [ ] On file select or drop: disable upload button, show inline spinner next to filename.
- [ ] Success: brief green checkmark flash (300ms) on the new file entry, then settle into normal state. Focus moves to the new file in the list.
- [ ] Error (wrong type): toast below sidebar header: "Unsupported file type. Accepted: .md, .txt, .json, .yml" with dismiss after 4s.
- [ ] Error (too large): toast: "File exceeds 500KB limit" with dismiss after 4s.
- [ ] Error (server): toast: "Upload failed. Try again." with a Retry action button. Auto-dismiss after 5s.

#### Delete confirmation

- [ ] Delete button click opens an inline confirmation (not a modal): "Delete {filename}?" with Cancel (default focus) and Delete (red/danger styled) buttons.
- [ ] After delete: focus moves to the next file in the list, or to the empty state if none remain.

#### Accessibility

- [ ] All icon buttons (upload, edit, delete, close, toggle sidebar) have `aria-label` attributes.
- [ ] Sidebar has `role="complementary"` with `aria-label="Knowledge base"`.
- [ ] Editor panel has focus trap when open; Escape closes it (with unsaved changes guard).
- [ ] @ mention dropdown: `role="listbox"` with `aria-activedescendant` for the highlighted item. Each item has `role="option"`.
- [ ] File chips in chat input: each chip has an `aria-label` ("Remove {filename}") on the dismiss button.
- [ ] Delete confirmation uses `role="alertdialog"` with `aria-describedby` pointing to the confirmation text.
- [ ] Scope toggle (project/conversation) uses `role="tablist"` with `aria-selected` state.

## Key Patterns

### KnowledgeFile entity

```python
@dataclass
class KnowledgeFile:
    name: str
    description: str
    content: str
    file_type: str                          # "md" | "txt" | "json" | "yml"
    scope: str                              # "project" | "conversation"
    tags: list[str] = field(default_factory=list)
    id: str = field(default_factory=lambda: str(uuid4()))
    conversation_id: str | None = None
    created_at: datetime = field(default_factory=datetime.utcnow)
    updated_at: datetime = field(default_factory=datetime.utcnow)
```

### Catalog injection (system prompt section)

```
## Knowledge Base

The following files are available for reference. Use `read_knowledge_file` to retrieve full contents.

- **api-spec** (json, project): OpenAPI specification for the payments service [id: abc-123]
- **style-guide** (md, project): Frontend component naming and structure conventions [id: def-456]
- **meeting-notes** (txt, conversation): Notes from April 9 architecture review [id: ghi-789]
```

### Frontmatter auto-generation heuristics

| File type | Name source                        | Description source                                    | Tags source            |
| --------- | ---------------------------------- | ----------------------------------------------------- | ---------------------- |
| `.md`     | First `#` heading or filename      | First non-heading paragraph (truncated to ~200 chars) | Headings as topic tags |
| `.txt`    | Filename                           | First non-empty line (truncated)                      | `["text"]`             |
| `.json`   | Filename or top-level `"name"` key | Top-level keys summary                                | Top-level keys as tags |
| `.yml`    | Filename or top-level `name` key   | Top-level keys summary                                | Top-level keys as tags |

### Tool definition

```python
@tool
def read_knowledge_file(file_id: str) -> str:
    """Read full contents of a knowledge base file by ID.

    Use when you need detailed information from a file listed in the
    knowledge base catalog. The catalog shows file name, description,
    and tags to help you decide which files are relevant.
    """
```

## Agent Execution Strategy

### Parallelism map

| Phase | Depends on | Agent type | Notes                                                   |
| ----- | ---------- | ---------- | ------------------------------------------------------- |
| 1     | None       | backend    | Domain + persistence, can start immediately             |
| 2     | Phase 1    | backend    | Needs KnowledgeFile entity and repository               |
| 3     | Phase 2    | backend    | Needs KnowledgeService for catalog                      |
| 4     | 2, 3       | backend    | Needs service + agent tool layers                       |
| 5     | Phase 4    | frontend   | Needs REST API endpoints                                |
| 6     | Phase 4    | frontend   | Needs REST API endpoints, can run parallel with Phase 5 |
| 7     | None       | frontend   | UI/UX specs, runs parallel with Phases 5-6              |

### Sequencing constraints

- Phases 1 through 4 are sequential: each backend layer depends on the one below it.
- Phases 5, 6, and 7 can all run in parallel once Phase 4 (REST API) is complete.
- Phase 7 (UI/UX specs) has no code dependencies and can start immediately as a reference for Phases 5-6.

### Agent instructions

- **Backend agent (Phases 1-2)**: Follow PofEAA patterns. Entity in `domain/entities/`, protocol in `domain/repositories/`, SQLAlchemy model in `infrastructure/models/`, data mapper in `infrastructure/mappers/`. Use JSON array column for tags (same as `images_json` on `MessageModel`). Service uses heuristic-based frontmatter generation, not LLM.
- **Backend agent (Phases 3-4)**: Create LangChain `@tool` for file reading. Mirror the skills loader progressive disclosure pattern: catalog in prompt, full content on demand. DTOs use `alias_generator = to_camel`. Wire into DI in `main.py`.
- **Frontend agent (Phases 5-6)**: Build sidebar + editor components. Use Lucide icons. Implement `@` mention dropdown with fuzzy matching. Follow accessibility specs from Phase 7.
- **Frontend agent (Phase 7)**: Define UI/UX specs as a reference document. No code output, just detailed specifications for layout, empty states, feedback flows, accessibility, and motion.

## Open Questions

1. **Max file size limit?** Suggestion: 500KB per file (large enough for substantial docs, small enough to keep DB reasonable). Needs confirmation.
1. **Max files per scope?** Suggestion: 50 project-level, 20 per conversation. Prevents catalog bloat.
1. **LLM-powered frontmatter?** Phase 2 uses heuristics. A follow-up could use the LLM itself to generate better descriptions and tags, at the cost of latency on upload.
1. **Search within files?** Current design relies on the agent reading full files. A future phase could add keyword search across file contents as an additional tool.
